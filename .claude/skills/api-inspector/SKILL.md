# LifeStack API Inspector Toolkit

## Pre-flight (always do first)

Before any test action, verify the server is running:

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:8000/up
```

If not 200, tell the user to start the server.

## Modes of Operation

Determine the mode from the user's request, then follow the corresponding workflow.

### Mode 1: Test New/Modified Endpoint

When the user says "test this new endpoint", "verify this API works", or just developed a new endpoint.

**Workflow:**

1. **Read the source code** for the endpoint:
   - Route definition in `routes/api.php`
   - Controller method
   - FormRequest (validation rules)
   - Service method (business logic)
   - Resource (response shape)
   - Related Model and Enum if applicable

2. **Check OpenAPI spec** — fetch the spec and verify the endpoint appears:
   ```bash
   curl -s http://localhost:8000/docs/api.json | python3 -c "
   import json,sys; d=json.load(sys.stdin)
   path='/v1/ENDPOINT_PATH'
   if path in d['paths']:
       print(json.dumps(d['paths'][path], indent=2))
   else:
       print(f'NOT FOUND in spec: {path}')
   "
   ```

3. **Build test data** from the FormRequest rules (required fields, enum values, types)

4. **Get auth token** (if endpoint requires auth):
   ```bash
   TOKEN=$(curl -s -X POST http://localhost:8000/api/v1/auth/login \
     -H "Content-Type: application/json" \
     -d '{"email":"EMAIL","password":"PASSWORD"}' \
     | python3 -c "import json,sys; print(json.load(sys.stdin)['data']['token'])")
   ```
   If no test user exists, register one first with a timestamped email.

5. **Call the endpoint** with valid data and verify:
   - HTTP status code matches expected
   - Response body structure matches the Resource class
   - All required fields are present and correctly typed
   - Enum values are valid
   - Relationships load correctly

6. **Test error cases**:
   - Missing required fields (expect 422)
   - Invalid enum values (expect 422)
   - Non-existent resource ID (expect 404)
   - Without auth token (expect 401)

7. **Compare actual response with OpenAPI schema** — check if the auto-generated spec accurately describes the real response. Report any discrepancies.

8. **Clean up** test data created during testing.

9. Report results using the **Mode 1 — Endpoint Test Report** format from Output Report Formats below.

### Mode 2: Debug Endpoint

When the user says "debug this api", "this endpoint returns wrong data", or reports an API error.

**Workflow:**

1. **Reproduce the issue** — ask the user for the exact curl command or request details, then execute it

2. **Read the full request chain**:
   - `routes/api.php` — confirm route exists and middleware
   - Controller method — check authorization, service call
   - FormRequest — check validation rules match the sent data
   - Service method — trace the business logic
   - Model — check scopes, casts, relationships, `$fillable`
   - Policy — check authorization logic

3. **Check Laravel logs** for errors:
   ```bash
   tail -50 /Users/chuongle/tools/LifeStack/api/storage/logs/laravel.log
   ```

4. **Test variations** of the request to isolate the issue:
   - Minimal valid request (only required fields)
   - With each optional field added one at a time
   - With different enum values
   - With/without auth

5. Report results using the **Mode 2 — Debug Report** format from Output Report Formats below.

### Mode 3: Verify API Matches Documentation

When the user says "check if api matches docs", "verify api spec", or "are docs up to date".

**Workflow:**

1. **Fetch the live OpenAPI spec**:
   ```bash
   curl -s http://localhost:8000/docs/api.json > /tmp/lifestack-spec.json
   ```

2. **For each endpoint** (or a specific module if requested):
   - Read the OpenAPI schema for request body and response
   - Read the corresponding FormRequest (actual validation rules)
   - Read the corresponding Resource (actual response fields)
   - Call the endpoint and capture the actual response

3. **Compare three sources**:
   - OpenAPI spec (what docs say)
   - FormRequest + Resource (what code defines)
   - Actual response (what API returns)

4. **If out of sync**: Scramble generates docs from code, so if spec doesn't match code, the issue is likely missing PHPDoc annotations or Scramble not inferring correctly. Suggest fixes.

5. Report results using the **Mode 3 — Docs Verification Report** format from Output Report Formats below.

### Mode 4: Test API Workflow

When the user says "test workflow", "test flow", or wants to verify connections between APIs.

**Workflow:**

1. **Identify the workflow** from the user's request. Common workflows:

   - **Auth flow**: register -> login -> me -> update profile -> refresh -> logout
   - **Project lifecycle**: create project -> add tasks -> update task status -> check project task counts -> archive project
   - **Task lifecycle**: create project -> create task -> set as daily focus -> mark done -> check dashboard
   - **Idea pipeline**: create idea -> update idea -> convert to project -> verify project created + idea status changed
   - **Daily planning**: create tasks -> set daily focus (up to 3) -> reorder -> check dashboard today
   - **Full daily workflow**: login -> check dashboard -> create tasks -> set focus -> work (update status) -> review dashboard

2. **Build the chain** — each step uses data from previous steps (IDs, tokens, etc.)

3. **Execute sequentially**, verifying at each step:
   - The response is correct
   - Side effects from previous steps are reflected (e.g., task count updates after creating tasks)
   - Cross-module data is consistent (e.g., dashboard reflects task changes)

4. Report results using the **Mode 4 — Workflow Test Report** format from Output Report Formats below.

### Mode 5: Regression Suite

When the user says "run all tests", "regression", or "full test suite".

Run the test script:

```bash
bash /Users/chuongle/tools/LifeStack/.claude/skills/api-inspector/scripts/run_tests.sh [module]
```

Modules: `all`, `auth`, `projects`, `tasks`, `daily-focus`, `ideas`, `dashboard`

If tests fail, automatically switch to **Mode 2 (Debug)** for each failing endpoint.

Report results using the **Mode 5 — Regression Report** format from Output Report Formats below.

## Curl Conventions

- Always use `-s` (silent) to suppress progress bars
- Use `-w "\n%{http_code}"` to capture status codes
- Use `python3 -c "import json,sys; ..."` for JSON parsing (jq may not be installed)
- Use `[TEST]` prefix in titles for test data identification
- Use timestamped emails: `test_$(date +%s)@lifestack.test`
- Never include line breaks in Authorization header values
- Clean up all test data after testing
